1.1 Kiểm thử phần mềm là gì?

Kiểm thử phần mềm là quá trình chạy một chương trình hoặc ứng dụng với mục đích tìm ra lỗi, đảm bảo chương trình hoạt động đúng như kỳ vọng và đáp ứng yêu cầu của người dùng.

Đơn giản hơn: bạn viết code để kiểm tra code, thay vì tự tay bấm vào trình duyệt mỗi lần thay đổi.

Tại sao phải test?
  • Phát hiện lỗi sớm — khi sửa rẻ hơn nhiều lần so với khi đã production.
  • Tự tin refactor và thêm tính năng mới mà không sợ "vỡ" code cũ.
  • Tài liệu sống: test case giải thích code làm gì tốt hơn comment.
  • Cần thiết để CI/CD chạy được — pipeline không có test thì chỉ là auto-deploy, không phải CI/CD thật sự.

1.2 Manual test vs Automated test

Tiêu chíManual TestAutomated Test
Tốc độChậm, phụ thuộc ngườiNhanh, chạy trong giây
Độ lặp lạiDễ bỏ sót bước khi mệtLuôn nhất quán 100%
Chi phí ban đầuThấpCần thời gian viết test
Chi phí dài hạnTăng theo dự ánGiảm dần, ROI cao
Phù hợpExploratory, UX phức tạpRegression, smoke, unit

Trong thực tế, cả hai bổ sung nhau — không phải thay thế nhau. Automated test giải phóng tester khỏi việc click đi click lại để tập trung vào exploratory testing sáng tạo hơn.

1.3 Các loại test phổ biến

  • Unit test: Test một hàm / module nhỏ, độc lập. Nhanh nhất, nhiều nhất trong test pyramid.
  • Integration test: Test nhiều module kết hợp với nhau (ví dụ: hàm gọi API + xử lý dữ liệu).
  • E2E test (end-to-end): Mô phỏng người dùng thật: mở trình duyệt, điền form, nhấn nút, kiểm tra kết quả.
  • Smoke test: Chạy nhanh một vài test quan trọng nhất để xác nhận build không bị hỏng hoàn toàn.
  • Regression test: Chạy lại toàn bộ test sau khi thay đổi để đảm bảo không có gì bị vỡ.
Gợi nhớ nhanh: Unit = một hàm, Integration = nhiều hàm, E2E = cả người dùng.
