Phần 1: Nền tảng kiểm thử tự động

1.1. Kiểm thử phần mềm là gì?

Kiểm thử phần mềm là quá trình xác minh rằng phần mềm hoạt động đúng như mong đợi và phát hiện lỗi trước khi người dùng thật gặp lỗi đó.

Hiểu đơn giản: code là xây nhà, testing là kiểm tra cửa có đóng được không, điện có chập không, nước có rò không, trước khi bàn giao cho khách.

1.2. Có những loại test nào?

  • Manual test: con người tự click, nhập, kiểm tra kết quả.
  • Automated test: dùng code để tự kiểm tra phần mềm.
  • Unit test: test đơn vị nhỏ nhất như hàm, module.
  • Integration/Business test: test các phần phối hợp với nhau, test quy tắc nghiệp vụ.
  • E2E test: test luồng hoàn chỉnh như người dùng thật.
  • Smoke test: test nhanh các chức năng sống còn sau mỗi bản build.
  • Regression test: test lại để chắc rằng sửa chỗ này không làm hỏng chỗ khác.

1.3. Test pyramid là gì?

Test pyramid là cách cân bằng số lượng test:

  • Nhiều unit test ở đáy vì nhanh, rẻ, dễ bảo trì.
  • Ít hơn ở integration/business test.
  • Ít nhất ở E2E test vì chậm và dễ flaky.

1.4. CI là gì? CD là gì?

  • CI - Continuous Integration: mỗi lần push code, hệ thống tự chạy build, lint, test.
  • CD - Continuous Delivery/Deployment: sau khi qua kiểm tra, hệ thống chuẩn bị hoặc tự động deploy.

1.5. AI giúp gì trong testing?

  • Gợi ý test case từ user story.
  • Sinh khung unit test, E2E test.
  • Phân tích lỗ hổng coverage còn thiếu.
  • Tóm tắt log lỗi CI để debug nhanh hơn.

Nhưng AI không thay thế tư duy kiểm thử. Bạn vẫn phải tự kiểm tra yêu cầu, dữ liệu biên, luật nghiệp vụ, độ đúng sai của test.

2.4. Cài công cụ test

npm init -y
npm install -D vitest @playwright/test http-server
npx playwright install

Giải thích nhanh:

  • vitest: chạy unit test và business test nhanh.
  • @playwright/test: chạy E2E test trên trình duyệt thật.
  • http-server: mở local server để Playwright test được giao diện.

2.5. Thêm script vào package.json

{
  "scripts": {
    "start": "http-server ./src -p 3000",
    "test:unit": "vitest run tests/unit",
    "test:business": "vitest run tests/business",
    "test:e2e": "playwright test",
    "test": "npm run test:unit && npm run test:business && npm run test:e2e"
  }
}

2.6. Nguyên tắc setup dễ test

  • Tách logic tính toán khỏi code thao tác DOM.
  • Hàm nên nhận input rõ ràng và trả output rõ ràng.
  • Đặt selector ổn định cho E2E, ví dụ data-testid.
  • Không nhét toàn bộ logic vào một file app.js.

Phần 3: Viết tài liệu test plan và test case

3.1. Trước khi viết code test, hãy viết test case

Test case giúp bạn biết mình cần kiểm cái gì. Nếu chưa có test case, bạn rất dễ viết test theo cảm tính và bỏ sót tình huống quan trọng.

3.2. Một test case thường gồm gì?

  • Mã test case.
  • Mục tiêu kiểm thử.
  • Điều kiện đầu vào.
  • Các bước thực hiện.
  • Kết quả mong đợi.
  • Mức độ ưu tiên.

3.3. Ví dụ test case cho form đặt hàng

ID Tình huống Bước thực hiện Kết quả mong đợi
TC-01 Nhập hợp lệ toàn bộ form Nhập tên, số lượng = 2, coupon = SALE10, bấm Tính tiền Hiển thị tổng tiền sau giảm giá chính xác
TC-02 Bỏ trống họ tên Không nhập tên, bấm Tính tiền Hiển thị lỗi "Họ tên là bắt buộc"
TC-03 Số lượng bằng 0 Nhập quantity = 0 Không cho submit, báo lỗi số lượng phải lớn hơn 0
TC-04 Mã giảm giá không hợp lệ Nhập coupon = FAKE999 Không áp giảm giá, có thông báo coupon không hợp lệ
TC-05 Kiểm thử dữ liệu biên Nhập quantity = 999 Hệ thống xử lý đúng hoặc chặn theo rule đã định

3.4. Kỹ thuật thiết kế test case cần biết

  • Happy path: luồng đúng, người dùng nhập chuẩn.
  • Negative case: nhập sai, thiếu dữ liệu, dữ liệu lạ.
  • Boundary value: min, max, ngay sát min/max.
  • Equivalence partitioning: chia input thành nhóm hợp lệ/không hợp lệ.
  • Error guessing: đoán lỗi từ kinh nghiệm thực tế.

Phần 4: Viết unit test

4.1. Tách hàm để dễ test

// src/js/validation.js
export function validateCheckoutForm({ customerName, quantity }) {
  const errors = [];

  if (!customerName || customerName.trim() === "") {
    errors.push("Họ tên là bắt buộc");
  }

  if (!Number.isInteger(quantity) || quantity <= 0) {
    errors.push("Số lượng phải lớn hơn 0");
  }

  return errors;
}

4.2. Viết unit test bằng Vitest

// tests/unit/validation.test.js
import { describe, expect, it } from "vitest";
import { validateCheckoutForm } from "../../src/js/validation.js";

describe("validateCheckoutForm", () => {
  it("trả về lỗi khi thiếu họ tên", () => {
    const errors = validateCheckoutForm({ customerName: "", quantity: 1 });
    expect(errors).toContain("Họ tên là bắt buộc");
  });

  it("trả về lỗi khi số lượng không hợp lệ", () => {
    const errors = validateCheckoutForm({ customerName: "An", quantity: 0 });
    expect(errors).toContain("Số lượng phải lớn hơn 0");
  });

  it("không có lỗi khi dữ liệu hợp lệ", () => {
    const errors = validateCheckoutForm({ customerName: "An", quantity: 2 });
    expect(errors).toEqual([]);
  });
});

4.3. Test coverage nên nhìn thế nào?

  • Không chạy theo 100% coverage một cách mù quáng.
  • Ưu tiên test các hàm có nghiệp vụ, tính toán, điều kiện rẽ nhánh.
  • Các file chỉ chứa hằng số hoặc wiring mỏng có thể ít quan trọng hơn.

4.4. Lỗi người mới hay gặp

  • Test cả DOM phức tạp trong unit test.
  • Viết test phụ thuộc thứ tự chạy.
  • Dùng dữ liệu test khó hiểu, không nói lên mục tiêu kiểm thử.
  • Không test case biên.

Phần 5: Viết test nghiệp vụ (business test)

5.1. Business test là gì?

Đây là loại test tập trung vào quy tắc nghiệp vụ, ví dụ: giảm giá 10% cho đơn từ 2 sản phẩm, miễn phí vận chuyển từ 500.000đ, không cho dùng đồng thời 2 coupon.

5.2. Ví dụ hàm nghiệp vụ

// src/js/pricing.js
export function calculateTotal({ quantity, unitPrice, couponCode }) {
  const subtotal = quantity * unitPrice;
  let discount = 0;

  if (couponCode === "SALE10") {
    discount = subtotal * 0.1;
  }

  return subtotal - discount;
}

5.3. Business test

// tests/business/checkout.test.js
import { describe, expect, it } from "vitest";
import { calculateTotal } from "../../src/js/pricing.js";

describe("calculateTotal", () => {
  it("giảm 10% khi dùng mã SALE10", () => {
    const total = calculateTotal({ quantity: 2, unitPrice: 100000, couponCode: "SALE10" });
    expect(total).toBe(180000);
  });

  it("không giảm giá khi coupon không hợp lệ", () => {
    const total = calculateTotal({ quantity: 2, unitPrice: 100000, couponCode: "ABC" });
    expect(total).toBe(200000);
  });
});

5.4. Khác nhau giữa unit test và business test

  • Unit test: tập trung vào một hàm/đơn vị nhỏ.
  • Business test: tập trung kiểm tra đúng sai của luật nghiệp vụ.

Trong dự án nhỏ, hai loại này có thể dùng chung framework nhưng khác mục tiêu.

Phần 6: Viết test E2E với Playwright

6.1. Khi nào cần E2E?

  • Khi muốn test luồng người dùng thật từ đầu đến cuối.
  • Khi cần chắc rằng HTML + CSS + JS + API + điều hướng hoạt động cùng nhau.
  • Khi muốn bảo vệ các luồng quan trọng như đăng nhập, thanh toán, đặt hàng.

6.2. Cấu hình Playwright tối thiểu

// playwright.config.js
import { defineConfig } from "@playwright/test";

export default defineConfig({
  testDir: "./tests/e2e",
  use: {
    baseURL: "http://127.0.0.1:3000",
    headless: true
  },
  webServer: {
    command: "npm run start",
    url: "http://127.0.0.1:3000",
    reuseExistingServer: true
  }
});

6.3. Ví dụ E2E test

// tests/e2e/checkout.spec.js
import { expect, test } from "@playwright/test";

test("người dùng nhập hợp lệ và xem được tổng tiền", async ({ page }) => {
  await page.goto("/");
  await page.getByLabel("Họ tên").fill("Nguyễn Văn A");
  await page.getByLabel("Số lượng").fill("2");
  await page.getByLabel("Mã giảm giá").fill("SALE10");
  await page.getByRole("button", { name: "Tính tiền" }).click();

  await expect(page.locator("#result")).toContainText("180000");
});

6.4. Mẹo viết E2E ổn định

  • Dùng selector có ý nghĩa như label, role, test id.
  • Không phụ thuộc vào CSS selector quá dài.
  • Tránh waitForTimeout nếu không cần thiết.
  • Dữ liệu test phải dễ reset, độc lập giữa các test.

Phần 7: Dùng AI để tăng tốc kiểm thử

7.1. AI nên được dùng ở đâu?

  • Phân rã yêu cầu thành test scenario.
  • Tạo khung test code ban đầu.
  • Đề xuất dữ liệu biên và negative case.
  • Review những case còn thiếu trong file test hiện tại.

7.2. Prompt mẫu để sinh test case

Hãy đóng vai QA Engineer senior.
Phân tích form đặt hàng gồm: họ tên bắt buộc, số lượng > 0, coupon SALE10 giảm 10%.
Hãy liệt kê:
1. Happy path
2. Negative case
3. Boundary case
4. Risk cần chú ý
Trình bày theo bảng test case.

7.3. Prompt mẫu để sinh unit test

Hãy viết unit test Vitest cho hàm calculateTotal.
Yêu cầu:
- Bao phủ coupon hợp lệ và không hợp lệ
- Có case quantity = 1, 2, 999
- Không mock gì cả
- Trả lời bằng code hoàn chỉnh

7.4. Checklist khi dùng AI

  • AI có hiểu đúng yêu cầu nghiệp vụ không?
  • Test sinh ra có assert đúng điều cần kiểm không?
  • Có bỏ sót case biên hoặc luồng lỗi không?
  • Tên test có mô tả rõ mục tiêu không?
  • Selector trong E2E có ổn định không?

Phần 8: Thiết lập CI/CD với GitHub Actions

8.1. Mục tiêu pipeline

  1. Mỗi lần push hoặc tạo pull request, hệ thống tự cài dependency.
  2. Tự chạy unit test và business test.
  3. Tự chạy E2E test.
  4. Nếu tất cả đều xanh, mới cho phép merge hoặc deploy.

8.2. Workflow mẫu

name: frontend-testing

on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout source
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Install Playwright browsers
        run: npx playwright install --with-deps

      - name: Run unit tests
        run: npm run test:unit

      - name: Run business tests
        run: npm run test:business

      - name: Run E2E tests
        run: npm run test:e2e

8.3. CD đơn giản nên hiểu thế nào?

  • Continuous Delivery: hệ thống chuẩn bị sẵn bản build, chờ người bấm deploy.
  • Continuous Deployment: qua test là tự deploy luôn.

8.4. Gợi ý pipeline nâng cao

  • Thêm lint và format check.
  • Chạy test matrix nhiều phiên bản Node.js.
  • Upload Playwright report làm artifact.
  • Tách job theo mức độ: lint - unit - business - e2e - deploy.

Phần 9: Kiến thức nâng cao cần học tiếp

  • Mocking: giả lập API hoặc dependency ngoài để test nhanh và ổn định.
  • Test doubles: stub, spy, mock, fake.
  • Flaky test: test lúc pass lúc fail, cần tìm nguyên nhân timing hoặc dữ liệu.
  • Coverage report: xem phần nào của code chưa được chạm tới.
  • Visual regression: so sánh ảnh chụp giao diện trước/sau.
  • Performance testing: đo tốc độ phản hồi và tải trang.
  • Security testing: kiểm tra XSS, CSRF, input injection, secret leak.

Phần 10: Bài tập thực hành

Bài tập 1 - Cơ bản

  • Tạo form đăng ký học viên bằng HTML CSS JS.
  • Viết tối thiểu 10 test case manual.
  • Viết unit test cho hàm validate email, validate số điện thoại.

Bài tập 2 - Trung cấp

  • Tạo trang giỏ hàng với subtotal, discount, shipping fee.
  • Viết business test cho các rule giảm giá.
  • Viết 3 E2E test cho luồng thêm sản phẩm, cập nhật số lượng, thanh toán.

Bài tập 3 - Nâng cao

  • Tự cấu hình GitHub Actions để chạy toàn bộ test khi push code.
  • Dùng AI để đề xuất thêm test case bị thiếu.
  • So sánh đề xuất của AI với test case do bạn tự nghĩ ra.

Đề bài review cuối khóa

  1. Source có tách logic đủ để unit test chưa?
  2. Test case có bao phủ happy path, negative, boundary chưa?
  3. E2E có tập trung vào luồng quan trọng nhất chưa?
  4. CI có chặn merge khi test fail chưa?
  5. AI đang hỗ trợ bạn hay đang làm bạn chủ quan?

Tóm tắt cách học hiệu quả

  1. Hiểu yêu cầu nghiệp vụ trước, đừng nhảy vào viết test ngay.
  2. Viết test case giấy/trên tài liệu trước khi viết code test.
  3. Tách logic nhỏ để unit test dễ.
  4. Chỉ chọn vài luồng E2E thật quan trọng để giảm chi phí bảo trì.
  5. Dùng AI để tăng tốc, nhưng luôn review lại bằng tư duy của chính bạn.
  6. Đưa tất cả vào CI/CD để team nào cũng được hưởng lợi từ tự động hóa.
