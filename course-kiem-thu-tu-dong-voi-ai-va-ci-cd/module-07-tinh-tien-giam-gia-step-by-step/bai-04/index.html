7.4.1 Mục tiêu bài học

Viết tài liệu trước để học viên hiểu “mình test cái gì” trước khi viết code test.

7.4.2 File cần sửa: docs/test-plan.md

Bảng test case bắt buộc có cột phân loại:

  • Happy path: luồng hợp lệ đi đúng.
  • Negative case: dữ liệu sai, thiếu.
  • Boundary case: giá trị biên như 0, 100, 101.
| Mã TC | Cấp độ | Phân loại | Mục tiêu | Dữ liệu vào | Kết quả mong đợi |
|---|---|---|---|---|---|
| Mã TC-UNIT-10 | Unit | Boundary case | Chấp nhận % giảm tại biên | "0", "100" | Không lỗi |

Giải thích: cột phân loại giúp học viên tự kiểm tra độ phủ test, tránh chỉ viết Happy path.

7.4.3 Quy tắc mapping

Mỗi test code đặt tên chứa đúng mã TC. Ví dụ: it("Mã TC-UNIT-10: ...").

7.4.4 Mẫu nội dung docs/test-plan.md

# Kế hoạch kiểm thử - Bài tập "Tính tiền giảm giá"

## 1. Mục tiêu
- Kiểm tra validate giá gốc.
- Kiểm tra validate phần trăm giảm giá.
- Kiểm tra tính đúng số tiền cần trả.
- Kiểm tra UI hiển thị đúng thông báo.

## 2. Phân loại test case
- Happy path: dữ liệu đúng, hệ thống tính tiền thành công.
- Negative case: dữ liệu thiếu hoặc sai kiểu.
- Boundary case: dữ liệu ở ranh giới như 0%, 100%, 101%.

## 3. Danh sách test case

| Mã TC | Cấp độ | Phân loại | Mục tiêu | Dữ liệu vào | Kết quả mong đợi |
|---|---|---|---|---|---|
| Mã TC-UNIT-01 | Unit | Negative case | Bỏ trống giá gốc | price = "" | Vui lòng nhập giá gốc! |
| Mã TC-UNIT-10 | Unit | Boundary case | Giảm giá ở biên | discount = "0", "100" | Không có lỗi |
| Mã TC-BIZ-07 | Business | Happy path | Tính tiền hợp lệ | "200000", "10" | Số tiền cần trả: 180000 VNĐ |
| Mã TC-E2E-04 | E2E | Happy path | UI tính đúng | Nhập form hợp lệ | Message success |

7.4.5 Cách học viên tự nghĩ thêm case

  1. Nhìn từng ô input và hỏi: bỏ trống thì sao?
  2. Nhìn từng rule và hỏi: nhỏ hơn rule thì sao, lớn hơn rule thì sao?
  3. Nhìn kết quả và hỏi: kết quả có cần làm tròn không?
  4. Nhìn UI và hỏi: khi lỗi thì class có đổi không?

7.4.6 Checklist tài liệu

  • Mỗi dòng có đúng một mã TC.
  • Có đủ 3 loại: Happy path, Negative case, Boundary case.
  • Dữ liệu vào viết cụ thể, không ghi chung chung.
  • Kết quả mong đợi phải đủ text để test code so sánh được.
